# Architecture ‚Äî Enterprise Fraud Detection API

A real-time fraud detection system that classifies financial transactions using a trained machine learning model and optionally generates human-readable explanations via a large language model. The system is containerized, self-hosted, and exposes a REST API over HTTPS.

## 3. Core Components

### 3.1 `main.py` ‚Äî Application Entry Point

Instantiates the FastAPI application, registers the router, and emits a structured startup log event on process launch. The startup event includes the active `model_version_id` so the deployed model version is traceable from logs.

```python
app = FastAPI(title="Enterprise Fraud Detection API")
app.include_router(router)
log_event("system_startup", {"model_version_id": settings.model_version_id})
```

### 3.2 `config.py` ‚Äî Configuration Layer

Reads all required environment variables at import time using Pydantic `BaseModel`. Raises `RuntimeError` immediately if any variable is missing. The application will not start with incomplete configuration ‚Äî there is no lazy loading.

**Required environment variables:**

| Variable | Type | Purpose |
|----------|------|---------|
| `DB_NAME` | str | PostgreSQL database name |
| `DB_USER` | str | PostgreSQL user |
| `DB_PASSWORD` | str | PostgreSQL password |
| `DB_HOST` | str | PostgreSQL host |
| `DB_PORT` | str | PostgreSQL port |
| `MODEL_VERSION_ID` | int | Which model version to load from `model_versions` |
| `OPENAI_API_KEY` | str | OpenAI API authentication |

### 3.3 `schemas.py` ‚Äî Request / Response Contracts

All API input and output shapes are defined as Pydantic v2 models. Validation runs at the FastAPI boundary before any application code executes.

**`TransactionInput`**
```
transaction_time_seconds : int
amount                   : float
v                        : List[float]  # exactly 28 elements enforced
```

The `v` field enforces `min_length=28, max_length=28`. Any request with a mismatched feature vector is rejected with HTTP 422 before reaching the ML layer.

**`PredictionResponse`**
```
fraud_probability : float   # model output [0.0, 1.0]
prediction        : int     # 0 = legitimate, 1 = fraud
threshold         : float   # decision threshold applied (0.8)
latency_ms        : float   # end-to-end request latency
```

**`PredictionWithExplanationResponse`** extends `PredictionResponse`:
```
explanation : str   # GPT-4o-mini generated natural language explanation
```

**`MetricsResponse`**
```
model_version_id            : int
total_predictions           : int
fraud_predictions           : int
fraud_rate                  : float
average_latency_ms          : float | None
llm_total_calls             : int
llm_total_prompt_tokens     : int
llm_total_completion_tokens : int
llm_total_tokens            : int
```

### 3.4 `routes.py` ‚Äî API Endpoints

Defines all four endpoints. `FraudModelService` and `ExplanationService` are instantiated **once at module load time** as process-level singletons and shared across all requests.

```python
model_service = FraudModelService()   # loaded once at startup
llm_service   = ExplanationService()  # loaded once at startup
```

**Endpoints:**

| Method | Path | Response Model | Description |
|--------|------|----------------|-------------|
| `GET` | `/health` | inline dict | Liveness check with active model version |
| `POST` | `/predict` | `PredictionResponse` | ML inference only |
| `POST` | `/predict-with-explanation` | `PredictionWithExplanationResponse` | ML + LLM |
| `GET` | `/metrics` | `MetricsResponse` | Aggregate stats from DB + in-memory LLM counters |

### 3.5 `ml.py` ‚Äî ML Inference Service

Owns all ML operations: model loading, integrity verification, feature engineering, and prediction execution.

**Startup sequence (runs once, blocks startup until complete):**

```
1. fetch_model_metadata(model_version_id)   ‚Üí DB query ‚Üí path, threshold, features
2. joblib.load(model_path)                  ‚Üí filesystem ‚Üí {pipeline, threshold, features}
3. _validate_model_integrity()
   ‚îú‚îÄ threshold mismatch  ‚Üí RuntimeError ‚Üí process exits
   ‚îî‚îÄ feature mismatch    ‚Üí RuntimeError ‚Üí process exits
4. Model ready to serve
```

**Per-request inference:**

```
transaction_time_seconds, amount, v[28]
        ‚îÇ
        ‚ñº
engineer_features()
  hour       = (transaction_time_seconds // 3600) % 24
  log_amount = log(amount) if amount != 0 else 0
  features   = [amount, hour, log_amount, v1..v28]  ‚Üí  shape (1, 31)
        ‚îÇ
        ‚ñº
StandardScaler.transform(features)
        ‚îÇ
        ‚ñº
LogisticRegression.predict_proba()[:, 1]  ‚Üí  prob: float
        ‚îÇ
        ‚ñº
int(prob >= 0.8)  ‚Üí  prediction: 0 or 1
```

**Model bundle structure (`.pkl`):**

```python
{
    "pipeline":  sklearn.pipeline.Pipeline([
                     ("scaler", StandardScaler()),
                     ("model",  LogisticRegression(
                                    class_weight="balanced",
                                    max_iter=2000,
                                    solver="lbfgs",
                                    penalty="l2",
                                    C=1.0
                                ))
                 ]),
    "threshold": 0.8,
    "features":  ["amount", "hour", "log_amount", "v1", ..., "v28"]  # 31 total
}
```

### 3.6 `llm.py` ‚Äî LLM Explanation Service

Wraps the OpenAI client and manages token usage tracking.

- **Model:** `gpt-4o-mini`
- **Temperature:** `0.2`
- **Token tracking:** in-memory per process ‚Äî resets on restart
- **Failure handling:** any `Exception` from the OpenAI API returns the static fallback string `"Explanation unavailable due to LLM service error."` ‚Äî the endpoint always returns HTTP 200 regardless of LLM health

### 3.7 `prompts.py` ‚Äî Prompt Construction

Builds the LLM prompt from the structured prediction result. Only three values are passed to the LLM: `fraud_probability`, `prediction`, and `threshold`. No raw transaction features are included.

```python
def fraud_explanation_prompt(structured_summary: dict) -> str:
    return f"""
You are explaining fraud detection results.

Rules:
- Do NOT change the decision.
- Do NOT invent new facts.
- Only explain the structured result.
- Keep explanation under 4 sentences.
- Be concise and factual.

Structured Result:
{structured_summary}
"""
```

### 3.8 `db.py` ‚Äî Database Access Layer

Raw `psycopg2` with a context manager pattern. Three functions:

| Function | SQL Operation | Called By |
|----------|--------------|-----------|
| `fetch_model_metadata(version_id)` | `SELECT` from `model_versions` | `ml.py` at startup |
| `log_prediction(...)` | `INSERT` into `predictions` | `routes.py` per request |
| `fetch_metrics(version_id)` | 3√ó `SELECT` from `predictions` | `routes.py` on `/metrics` |

`log_prediction()` wraps the INSERT in `try/except Exception: pass`. Audit log failures never propagate to the caller.

### 3.9 `logger.py` ‚Äî Structured Logging

Emits JSON to stdout on every system event. All records include `timestamp` (UTC ISO 8601) and `event_type`.

```json
{
  "timestamp": "2026-03-02T22:59:14.123456",
  "event_type": "prediction_made",
  "model_version_id": 1,
  "fraud_probability": 0.9987,
  "prediction": 1,
  "latency_ms": 634.2
}
```

**Logged events:** `system_startup`, `prediction_made`, `prediction_with_explanation`, `prediction_error`, `metrics_requested`

Stdout is the correct output target for containerized deployment ‚Äî the container runtime captures it and any log aggregator (Datadog, Loki, CloudWatch) consumes it from there.

## 6. Storage Layer

### 6.1 PostgreSQL Schema

**`model_versions`** ‚Äî model registry

```sql
CREATE TABLE model_versions (
    id                  SERIAL PRIMARY KEY,
    model_path          TEXT,       -- filesystem path to .pkl inside container
    threshold           NUMERIC,    -- decision threshold (0.8)
    feature_columns     TEXT,       -- comma-separated, 31 features
    training_fraud_rate NUMERIC
);
```

**`predictions`** ‚Äî immutable prediction audit log

```sql
CREATE TABLE predictions (
    id                SERIAL PRIMARY KEY,
    model_version_id  INTEGER,
    fraud_probability NUMERIC,
    prediction        INTEGER,    -- 0 or 1
    latency_ms        NUMERIC
);
```

**`transactions`** ‚Äî training data store

```sql
CREATE TABLE transactions (
    id                        SERIAL PRIMARY KEY,
    transaction_time_seconds  INTEGER,
    amount                    FLOAT,
    v1 .. v28                 FLOAT,
    is_fraud                  BOOLEAN
);
```

### 6.2 Connection Behavior

Each database operation opens a new TCP connection and closes it on completion. No connection pool exists anywhere in the codebase or dependency stack.

**Measured impact:** Mean latency increased 2.9√ó under 10-worker concurrency (663ms ‚Üí 1,917ms) due to connection establishment contention.

### 6.3 Model File Storage

The `.pkl` is on the local filesystem inside the container. The path is stored in `model_versions.model_path`. The `models/` directory is copied into the image at build time ‚Äî model updates require a full image rebuild.

### 6.4 In-Memory LLM State

Token counters in `ExplanationService` are process-scoped and reset on every restart. The `/metrics` endpoint returns a hybrid of durable DB stats and ephemeral in-memory LLM stats.

### 7.2 Feature Set

| Feature | Source | Engineering Applied |
|---------|--------|-------------------|
| `amount` | `amount` (API input) | none ‚Äî passed through |
| `hour` | `transaction_time_seconds` | `(seconds // 3600) % 24` |
| `log_amount` | `amount` | `log(amount)`; `0` if `amount == 0` |
| `v1`‚Äì`v28` | `v[0..27]` (API input) | none ‚Äî PCA already applied upstream |

### 7.3 Class Imbalance Handling

The creditcard dataset has a 0.17% fraud rate (~492 fraud in 284,807 transactions). `class_weight="balanced"` weights classes inversely proportional to frequency:

```
weight(fraud)  = 284,807 / (2 √ó 492)     ‚âà 289√ó
weight(legit)  = 284,807 / (2 √ó 284,315) ‚âà 0.5√ó
```

Each fraud case is weighted ~289√ó more than a legitimate case during training. This prevents the model from optimizing accuracy by predicting "legitimate" for everything.

### 7.4 Fraud Probability Distribution (5,000 real transactions)


### 8.2 LLM Input Boundary

The LLM receives exactly three values:

```python
{ "fraud_probability": float, "prediction": int, "threshold": float }
```

No raw transaction features, PCA components, amount, time, or account data are passed. PCA components have no interpretable meaning to a language model ‚Äî passing them would produce hallucinated explanations. The LLM translates the model's decision into plain language; it does not re-analyze the transaction.

### 8.3 Prompt Guardrails

| Rule | Purpose |
|------|---------|
| `Do NOT change the decision` | Prevents the LLM from contradicting the ML model's output |
| `Do NOT invent new facts` | Prevents hallucinated transaction details |
| `Only explain the structured result` | Restricts output to provided data |
| `Keep explanation under 4 sentences` | Enforces conciseness for analyst use |

## 9. Failure Modes

### 9.1 Startup Failures ‚Äî Process Refuses to Start

| Condition | Cause | Recovery |
|-----------|-------|----------|
| Missing environment variable | Env not injected | Set all 7 required env vars |
| `model_version_id` not in DB | Wrong ID or unseeded DB | Insert correct record into `model_versions` |

## 11. Model Performance

Evaluated against 5,000 real transactions (creditcard dataset, `random_state=42`). True fraud prevalence: 6 of 5,000 (0.12%).

| Metric | Value | Note |
|--------|-------|------|
| Accuracy | 99.26% | Misleading on imbalanced data |
| Precision | 10.26% | 4 true positives out of 39 fraud alerts |
| Recall | 66.67% | 4 of 6 real fraud cases caught |
| F1 Score | 17.78% | Reflects class imbalance |
| False Positives | 35 | Legitimate transactions incorrectly flagged |
| False Negatives | 2 | Real fraud transactions missed |

Accuracy is not the signal on this dataset. A model that predicts "legitimate" on every transaction would achieve 99.88% accuracy. The meaningful metrics are recall (fraud caught) and precision (alert quality).

## 13. Deployment

### 13.1 Dockerfile Summary

```dockerfile
FROM python:3.12-slim

RUN apt-get update && apt-get install -y build-essential libpq-dev

```

`libpq-dev` is required for `psycopg2-binary`. `requirements.txt` is copied before application code to leverage Docker layer caching.

### 13.2 Running the Container

```bash
docker build -t fraud-detection-api .

docker run -d \
  -p 8000:8000 \
  -e DB_NAME=fraud_detection \
  -e DB_USER=postgres \
  -e DB_PASSWORD=your_password \
  -e DB_HOST=your_db_host \
  -e DB_PORT=5432 \
  -e MODEL_VERSION_ID=1 \
  -e OPENAI_API_KEY=your_openai_key \
  fraud-detection-api
```

### 13.3 Model Update Process

```
1. Retrain:   python scripts/prepare_data.py
              ‚Üí writes fraud_model_{timestamp}.pkl to models/

2. Register:  INSERT INTO model_versions (model_path, threshold, feature_columns, training_fraud_rate)
              VALUES ('models/fraud_model_{timestamp}.pkl', 0.8, 'amount,hour,...', 0.0017);

3. Rebuild:   docker build -t fraud-detection-api .

4. Redeploy:  docker run ... -e MODEL_VERSION_ID={new_id} fraud-detection-api
```

## 14. Security Posture

| Area | Status | Notes |
|------|--------|-------|
| API authentication | ‚ùå None | All endpoints publicly accessible |
| Rate limiting | ‚ùå None | `/predict-with-explanation` incurs OpenAI cost per call |
| Input validation | ‚úÖ Pydantic | Type and length enforced at boundary |
| Secrets in serving code | ‚úÖ Env vars | `config.py` reads all credentials from environment |
| TLS | ‚úÖ HTTPS | External hostname via DuckDNS |
| LLM input surface | ‚úÖ Minimal | Only three numeric values reach the LLM |

**Required before any public-facing production deployment:**

1. Add bearer token or API key authentication via FastAPI `Depends()` and `HTTPBearer`
2. Move hardcoded credentials in `load_data.py` / `prepare_data.py` to environment variables ‚Äî consistent with the pattern in `config.py`
3. Add rate limiting to protect OpenAI cost exposure on `/predict-with-explanation`
